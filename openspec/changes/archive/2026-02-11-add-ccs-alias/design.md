## Context

当前 `cc-switch` 是一个 CLI 工具，通过 `package.json` 的 `bin` 配置定义命令入口点。用户需要输入完整的 `cc-switch` 命令来执行各种操作（切换提供商、添加配置等），对于频繁使用的场景显得冗长。

**当前实现**：
- `package.json` 中定义 `"bin": { "cc-switch": "./dist/index.js" }`
- 用户通过 `cc-switch <command>` 调用 CLI

**约束条件**：
- 必须保持向后兼容，现有 `cc-switch` 命令不能受影响
- 不能引入新的外部依赖
- 两个命令应完全等价，共享相同的代码逻辑

## Goals / Non-Goals

**Goals:**
- 提供 `ccs` 作为 `cc-switch` 的简短别名
- `ccs` 支持所有现有命令和参数（如 `ccs add`、`ccs use <id>` 等）
- 保持完全向后兼容
- 更新文档说明新的别名用法

**Non-Goals:**
- 不改变现有命令的功能或行为
- 不弃用或移除 `cc-switch` 命令
- 不添加额外的命令行选项或特性

## Decisions

### 决策 1: 使用 npm bin 多入口点机制

**选择**: 在 `package.json` 的 `bin` 配置中添加 `ccs` 指向同一个入口文件

```json
{
  "bin": {
    "cc-switch": "./dist/index.js",
    "ccs": "./dist/index.js"
  }
}
```

**理由**:
- npm 原生支持多个 bin 入口点指向同一个文件
- 无需额外代码，两个命令自动共享所有逻辑
- 安装后会自动创建两个符号链接
- 完全透明，用户体验最佳

**替代方案**:
- ❌ 创建单独的 `ccs.js` 文件调用 `index.js`：增加维护成本，不必要的间接层
- ❌ 在代码中检测命令名：增加复杂度，且无法影响 Commander.js 的帮助信息

### 决策 2: 文档更新策略

**选择**: 在 README 中同时展示两种用法，优先推荐 `ccs`

**理由**:
- 新用户更倾向于使用更短的命令
- 现有用户可以继续使用熟悉的 `cc-switch`
- 文档示例使用 `ccs` 可以引导用户采用更简洁的方式

## Risks / Trade-offs

### 风险 1: 命令名称冲突
**风险**: `ccs` 可能与用户系统中的其他命令冲突
**缓解**:
- `ccs` 是相对罕见的命令名
- npm 局部安装（项目内使用）不会影响全局命令
- 如果冲突，用户仍可使用完整的 `cc-switch` 命令

### Trade-off 1: 双重命令维护
**取舍**: 文档和示例需要维护两个命令名称
**影响**:
- 文档示例优先使用 `ccs`，但在必要时提及 `cc-switch` 仍可用
- 长期来看，社区可能自然收敛到使用 `ccs`

### Trade-off 2: 发布后无法重命名
**取舍**: 一旦发布，`ccs` 别名将成为公共 API 的一部分
**影响**:
- 需要长期维护这个别名
- 好处是简短易记，值得长期支持
